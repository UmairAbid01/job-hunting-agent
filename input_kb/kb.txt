Question Refinement Pattern

Module 3

Why it exists:

You ask a vague question; the model proposes a *better*
question that adds missing constraints/context, then asks if
you want to use it.

Core move:

• Always propose a refined version

• Offer a yes/no switch to use refined vs original

• Helps you notice missing constraints

Where it fits in FYPs

Use it as a reusable *prompt primitive* inside
your agent loop / RAG UI.

Question Refinement Pattern — Technical Example

Module 3

Example prompt (copy/paste)

From now on, whenever I ask a question:

1) Suggest a better version of my question that adds
missing constraints (data, scope, success metric).

2) Ask: “Use the better question? (Yes/No)”.

If I say Yes, answer the better question. If No, answer
my original question.

My first question: “How can I improve my FYP?”

Sample output (what “good” looks like)

Suggested question adds: domain, dataset, objective
metric, latency/budget constraints, and current baseline.

Then model asks: “Use better question?
(Yes/No)”

Mini‑task (do in class)

Pick your own FYP topic. Write a 1‑line vague question.

Run the pattern and compare the refined
question with your original.

Record: which constraint changed the
answer the most?

Cognitive Verifier Pattern

Module 3

Why it exists:

LLMs often do better when they first generate sub‑questions
that expose missing variables and then combine answers.

Core move:

• Generate N clarifying sub‑questions

• Answer each (using user inputs if needed)

• Synthesize final answer

Where it fits in FYPs

Use it as a reusable *prompt primitive* inside
your agent loop / RAG UI.

Cognitive Verifier Pattern — Technical Example

Module 3

Example prompt (copy/paste)

When you are asked a question, follow these rules:

- Generate 5 additional questions that would help more
accurately answer it.

- Ask me to answer them (one at a time).

- Combine the answers to produce the final response.

Question: “Estimate the cost + latency of adding RAG to
our FYP chatbot.”

Sample output (what “good” looks like)

Outputs 5 sub‑questions: corpus size, embedding model,
vector DB choice, query QPS, target latency.

After answers, provides a cost/latency
estimate with assumptions.

Mini‑task (do in class)

Use your FYP. Ask a hard planning question (compute,
cost, timeline).

Answer the 5 sub‑questions.

Check if final plan changed vs your first
intuition.

Audience Persona Pattern

Module 3

Why it exists:

Instead of changing the assistant persona, you change the
*audience persona* so the explanation matches the reader
(HoD, client, junior dev, etc.).

Core move:

• State: “Explain X to me.”

• State: “Assume I am Persona Y.”

• Model adapts vocabulary + depth + examples

Where it fits in FYPs

Use it as a reusable *prompt primitive* inside
your agent loop / RAG UI.

Audience Persona Pattern — Technical Example

Module 3

Example prompt (copy/paste)

Explain retrieval‑augmented generation (RAG) to me.

Assume I am a final‑year BSAI student building a capstone
chatbot and I care about:

- evaluation metrics (EM/F1, faithfulness)

- failure modes (hallucinations)

- deployment constraints (latency, cost)

Sample output (what “good” looks like)

Explains RAG with: index→retrieve→ground→generate
pipeline; metrics; common bugs; and a deployment
checklist.

Mini‑task (do in class)

Rewrite Persona Y for 3 audiences:

(1) Non‑technical client, (2) security
reviewer, (3) your supervisor.

Compare how the explanation changes.

Flipped Interaction Pattern

Module 3

Why it exists:

You want the model to drive the Q/A loop: it asks questions
until it has enough info to produce a plan/design/output.

Core move:

• Tell it the goal X

• Tell it the stopping condition Y

• Ask for first question; optionally one‑at‑a‑time

Where it fits in FYPs

Use it as a reusable *prompt primitive* inside
your agent loop / RAG UI.

Flipped Interaction Pattern — Technical Example

Module 3

Example prompt (copy/paste)

I would like you to ask me questions to help me design my
FYP chatbot architecture.

Ask questions until you have enough information to:

(1) propose a baseline architecture, and

(2) propose an evaluation plan.

Ask one question at a time. Ask me the first question.

Sample output (what “good” looks like)

Starts with: “What is the user + task?” then asks
data/privacy/tooling questions.

Stops with a 1‑page architecture + eval
plan.

Mini‑task (do in class)

Run this for your FYP. Do not answer with
essays—answer with short facts.

Goal: reach a complete architecture in <10
turns.

Save the final plan as your baseline
document.

Few‑Shot Prompting

Module 4

Why it exists:

Teach the model a mapping by showing a few input→output
examples. Useful when you need consistent labels/format.

Core move:

• Provide 2–5 examples

• Keep prefixes consistent (Input:, Output:)

• Then give a new input and let it continue the pattern

Where it fits in FYPs

Use it as a reusable *prompt primitive* inside
your agent loop / RAG UI.

Few‑Shot Prompting — Technical Example

Module 4

Example prompt (copy/paste)

Task: classify bug report severity as {S1,S2,S3}.

Input: “API down for all users, 500 errors”

Output: S1

Input: “One endpoint slow, workaround exists”

Output: S2

Input: “Minor UI glitch in settings page”

Output: S3

Input: “Vector DB query times out for most queries after
deployment”

Output:

Sample output (what “good” looks like)

Model continues: “S1” or “S2” depending on described
impact.

Key: format stays short and consistent.

Mini‑task (do in class)

Create a 3‑label classifier relevant to your FYP (e.g.,
“Hallucination / Partial / Grounded”).

Write 3 examples.

Test on 5 new cases and see consistency.

Few‑Shot for Actions

Module 4

Why it exists:

Few‑shot can teach *what action to take* given a situation.
Useful for tool‑using agents.

Core move:

• Use Situation: … / Action: … pairs

• Keep actions as imperative verbs

• Use it to generate more examples, then curate

Where it fits in FYPs

Use it as a reusable *prompt primitive* inside
your agent loop / RAG UI.

Action: use_tool(read_pdf)

Situation: user asks “compare two answers”

Action: use_tool(compare_outputs)

Situation: user asks “find missing citation”

Action: use_tool(web_search)

Situation: user says “my results look wrong”

Action:

Model continues with an action like:

Why it exists:

Add Think/Action/Observation steps so the model learns a

Core move:

Writing Effective Few‑Shot Examples

Module 4

Why it exists:

Few‑shot fails when examples are ambiguous, prefixes are
too generic, or new inputs lack required context.

Core move:

• Use descriptive prefixes (not just Input/Output)

• Cover edge cases

• Ensure new input contains enough info to decide

Where it fits in FYPs

Use it as a reusable *prompt primitive* inside
your agent loop / RAG UI.

Few‑Shot Quality Checklist

Module 4

1) If you gave the examples to 10 humans, would they agree on the mapping?

2) Are the prefixes meaningful (e.g., ‘Object:’, ‘Constraint:’, ‘Expected label:’)?

3) Are labels bounded (explicit set of allowed outputs)?

4) Does each new input include the variables needed to choose a label?

5) Did you include at least one tricky/edge case example?

6) After generation, curate the examples and re‑use the curated set.
